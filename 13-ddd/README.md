# 🏗️ ডোমেইন-ড্রিভেন ডিজাইন (Domain-Driven Design - DDD)

## 📌 ভূমিকা

**ডোমেইন-ড্রিভেন ডিজাইন (DDD)** হলো সফটওয়্যার ডেভেলপমেন্টের একটি পদ্ধতি যেখানে সফটওয়্যারের
ডিজাইন এবং আর্কিটেকচার সম্পূর্ণভাবে **বিজনেস ডোমেইনের** উপর ভিত্তি করে তৈরি হয়। ২০০৩ সালে
**Eric Evans** তাঁর বিখ্যাত বই *"Domain-Driven Design: Tackling Complexity in the Heart of Software"*
-এ এই ধারণাটি প্রথম উপস্থাপন করেন।

> 💡 সহজ ভাষায়: DDD বলে — "তোমার সফটওয়্যার কোড যেন তোমার বিজনেসের ভাষায় কথা বলে।"

### 🤔 কেন DDD দরকার?

ধরুন আপনি **Daraz Bangladesh**-এর মতো একটি বিশাল ই-কমার্স প্ল্যাটফর্ম তৈরি করছেন। এখানে —
- প্রোডাক্ট ক্যাটালগ আছে
- অর্ডার ম্যানেজমেন্ট আছে
- পেমেন্ট প্রসেসিং (bKash, Nagad) আছে
- ডেলিভারি ট্র্যাকিং (Pathao Courier) আছে
- কাস্টমার সাপোর্ট আছে

এত জটিল একটি সিস্টেমে যদি সবকিছু এক জায়গায় গুলিয়ে ফেলেন, তাহলে কোড মেইনটেইন করা অসম্ভব হয়ে
যাবে। DDD আপনাকে শেখায় কিভাবে এই জটিলতাকে ভেঙে ছোট ছোট অর্থপূর্ণ অংশে সাজাতে হয়।

## 🎯 কেন DDD শিখবেন?

### ✅ DDD শেখার সুবিধা

- 🧠 **বিজনেস লজিক পরিষ্কার থাকে** — কোড পড়লেই বোঝা যায় সিস্টেম কী করে
- 🔀 **জটিল সিস্টেম ভাগ করা সহজ** — Bounded Context দিয়ে বড় সিস্টেম ছোট ছোট অংশে ভাগ করা যায়
- 🤝 **ডেভেলপার ও বিজনেস টিমের মধ্যে সেতুবন্ধন** — Ubiquitous Language সবাইকে একই ভাষায় কথা বলতে সাহায্য করে
- 🏢 **মাইক্রোসার্ভিস আর্কিটেকচারের ভিত্তি** — প্রতিটি Bounded Context একটি মাইক্রোসার্ভিস হতে পারে
- 🔄 **পরিবর্তন সহজ হয়** — এক ডোমেইনে পরিবর্তন করলে অন্য ডোমেইনে প্রভাব পড়ে না
- 📈 **স্কেলেবিলিটি** — আলাদা আলাদা ডোমেইন স্বতন্ত্রভাবে স্কেল করা যায়
- 🧪 **টেস্টিং সহজ** — ডোমেইন লজিক আলাদা থাকায় ইউনিট টেস্ট লেখা সহজ

### ❌ DDD কখন ব্যবহার করবেন না

- 🚫 **ছোট/সাধারণ CRUD অ্যাপ্লিকেশন** — একটি সাধারণ ব্লগ বা Todo অ্যাপে DDD ওভারকিল
- 🚫 **বিজনেস লজিক কম থাকলে** — শুধু ডেটা ইন/আউট করলে DDD দরকার নেই
- 🚫 **ছোট টিম ও সীমিত সময়** — DDD শেখা ও প্রয়োগ করতে সময় লাগে
- 🚫 **প্রোটোটাইপ বা MVP** — দ্রুত কিছু বানাতে চাইলে DDD বোঝা ভারী হতে পারে

## 📊 DDD vs Traditional Development — তুলনামূলক বিশ্লেষণ

```
┌──────────────────────┬───────────────────────────┬───────────────────────────┐
│       বিষয়           │   Traditional (CRUD)      │   Domain-Driven Design    │

│ আর্কিটেকচার         │ Controller → Service →    │ Domain Model কেন্দ্রিক     │
│                      │ Repository (ডেটা কেন্দ্রিক) │ (বিজনেস লজিক কেন্দ্রিক)  │
├──────────────────────┼───────────────────────────┼───────────────────────────┤
│ নামকরণ              │ টেকনিক্যাল টার্ম ব্যবহার   │ বিজনেস টার্ম ব্যবহার      │

│                      │ জমা হয় (Fat Service)      │ এনক্যাপসুলেটেড            │
├──────────────────────┼───────────────────────────┼───────────────────────────┤
│ টিম কমিউনিকেশন      │ ডেভেলপাররা নিজেদের মতো    │ সবাই Ubiquitous Language  │
```

## 📐 DDD-র মূল ধারণাগুলো — সংক্ষিপ্ত পরিচিতি

### স্ট্র্যাটেজিক ডিজাইন (Strategic Design)

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   📣 Ubiquitous Language (সর্বজনীন ভাষা)                        │
│   ──────────────────────────────────────                         │
│   ডেভেলপার, বিজনেস অ্যানালিস্ট, ম্যানেজার — সবাই               │
│   একই শব্দ ব্যবহার করবে। যেমন bKash-এ "Send Money"             │
│   বলতে সবাই একই জিনিস বুঝবে।                                   │
│                                                                  │
│   🔲 Bounded Context (সীমাবদ্ধ প্রসঙ্গ)                          │
│   ──────────────────────────────────────                         │
│   প্রতিটি ডোমেইনের নিজস্ব সীমানা আছে। "Customer" শব্দটি        │
│   Order Context-এ এক অর্থ বহন করে, আবার Shipping                │
│   Context-এ আরেক অর্থ।                                          │
│                                                                  │
│   🗺️ Context Map (প্রসঙ্গ মানচিত্র)                              │
│   ──────────────────────────────────                             │
│   বিভিন্ন Bounded Context কিভাবে একে অপরের সাথে                │
│   যোগাযোগ করে — তার মানচিত্র।                                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### ট্যাক্টিক্যাল ডিজাইন (Tactical Design)

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   🆔 Entity (সত্তা)                                              │
│   ─────────────────                                              │
│   একটি ইউনিক আইডেন্টিটি আছে। যেমন: Order (অর্ডার নম্বর দিয়ে   │
│   চেনা যায়), Customer (ফোন নম্বর দিয়ে চেনা যায়)।               │
│                                                                  │
│   💎 Value Object (মান বস্তু)                                     │
│   ─────────────────────────                                      │
│   কোনো আইডেন্টিটি নেই, শুধু মান দিয়ে তুলনা হয়।                │
│   যেমন: Money(500, "BDT"), Address("ঢাকা", "১২০৫")              │
│                                                                  │
│   📦 Aggregate (সমষ্টি)                                           │
│   ────────────────────                                           │
│   Entity ও Value Object-এর একটি গ্রুপ যেটি একটি                 │
│   Aggregate Root দিয়ে নিয়ন্ত্রিত হয়।                           │
│   যেমন: Order (root) → OrderItem, ShippingAddress                │
│                                                                  │
│   🗄️ Repository (ভান্ডার)                                        │
│   ────────────────────────                                       │
│   Aggregate-কে ডেটাবেইজে সংরক্ষণ ও পুনরুদ্ধার করার              │
│   ইন্টারফেস।                                                     │
│                                                                  │
│   ⚙️ Domain Service (ডোমেইন সেবা)                                │
│   ────────────────────────────────                               │
│   এমন বিজনেস লজিক যা কোনো নির্দিষ্ট Entity-র মধ্যে             │
│   রাখা যায় না। যেমন: MoneyTransferService (bKash-এ              │
│   এক অ্যাকাউন্ট থেকে আরেক অ্যাকাউন্টে টাকা পাঠানো)            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## ✅ প্রাক-প্রয়োজনীয়তা (Prerequisites)

DDD শেখার আগে নিচের বিষয়গুলো সম্পর্কে ভালো ধারণা থাকা দরকার:

| # | বিষয় | কেন দরকার |
|---|-------|-----------|
| ১ | **OOP (অবজেক্ট ওরিয়েন্টেড প্রোগ্রামিং)** | DDD-র ভিত্তি হলো অবজেক্ট মডেলিং |
| ২ | **SOLID প্রিন্সিপলস** | ক্লিন ও মেইনটেইনেবল কোড লেখার জন্য |
| ৩ | **ডিজাইন প্যাটার্নস** | Repository, Factory, Strategy ইত্যাদি প্যাটার্ন DDD-তে ব্যবহৃত হয় |
| ৪ | **ক্লিন আর্কিটেকচার** | লেয়ার্ড আর্কিটেকচার বোঝা DDD বুঝতে সাহায্য করে |
| ৫ | **বেসিক সফটওয়্যার আর্কিটেকচার** | Monolith, Microservices, Event-driven সম্পর্কে ধারণা |
| ৬ | **ডেটাবেইজ ডিজাইন** | Entity-Relationship মডেলিং বোঝা দরকার |
