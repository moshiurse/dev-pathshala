## 📊 Kafka ক্লাস্টার আর্কিটেকচার

### 🏠 ক্লাস্টার কী?

Kafka কখনো একটা সিঙ্গেল সার্ভারে চলে না — প্রোডাকশনে এটি একাধিক সার্ভারের একটি **ক্লাস্টার** হিসেবে কাজ করে।
ধরুন bKash-এর পেমেন্ট সিস্টেমে প্রতি সেকেন্ডে হাজার হাজার ট্রানজ্যাকশন আসছে। একটা সার্ভার দিয়ে
এটা হ্যান্ডেল করা অসম্ভব। তাই Kafka কে একাধিক **Broker** মিলে একটা ক্লাস্টার হিসেবে ডিপ্লয় করা হয়।

### 📌 ফুল ক্লাস্টার ডায়াগ্রাম

```
┌──────────────────────────────────────────────────────────────────────┐
│                        Kafka Cluster                                 │
│                                                                      │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐          │
│   │   Broker 1    │    │   Broker 2    │    │   Broker 3    │        │
│   │ (Controller)  │    │               │    │               │        │
│   │               │    │               │    │               │        │
│   │ ┌──────────┐ │    │ ┌──────────┐ │    │ ┌──────────┐ │        │
│   │ │ Topic A  │ │    │ │ Topic A  │ │    │ │ Topic A  │ │        │
│   │ │ P0 (L)   │ │    │ │ P0 (F)   │ │    │ │ P1 (L)   │ │        │
│   │ │ P1 (F)   │ │    │ │ P1 (F)   │ │    │ │ P0 (F)   │ │        │
│   │ └──────────┘ │    │ └──────────┘ │    │ └──────────┘ │        │
│   │ ┌──────────┐ │    │ ┌──────────┐ │    │ ┌──────────┐ │        │
│   │ │ Topic B  │ │    │ │ Topic B  │ │    │ │ Topic B  │ │        │
│   │ │ P0 (F)   │ │    │ │ P0 (L)   │ │    │ │ P0 (F)   │ │        │
│   │ └──────────┘ │    │ └──────────┘ │    │ └──────────┘ │        │
│   └──────┬───────┘    └──────┬───────┘    └──────┬───────┘        │
│          │                   │                   │                  │
│          └───────────────────┼───────────────────┘                  │
│                              │                                      │
│                    Inter-Broker Communication                       │
│                     (Replication Protocol)                           │
└──────────────────────────────────────────────────────────────────────┘
```

### 📌 Broker কী করে?

প্রতিটি Broker হলো একটি আলাদা Kafka সার্ভার প্রসেস। এর কাজ:

| দায়িত্ব | বিবরণ |
|----------|--------|
| **মেসেজ স্টোর** | প্রোডিউসারের পাঠানো মেসেজ ডিস্কে সংরক্ষণ করা |
| **মেসেজ সার্ভ** | কনজিউমারকে মেসেজ পড়তে দেওয়া |
| **রেপ্লিকেশন** | অন্য ব্রোকার থেকে ডাটা কপি করা (ফলোয়ার হিসেবে) |
| **পার্টিশন লিডার** | নির্দিষ্ট পার্টিশনের জন্য রিড/রাইট হ্যান্ডেল করা |

### 📌 Controller Broker

ক্লাস্টারের একটি ব্রোকার **Controller** হিসেবে নির্বাচিত হয়। Controller-এর বিশেষ দায়িত্ব:

```
┌─────────────────────────────────────────────┐
│            Controller Broker                 │
│                                             │
│  ✅ পার্টিশন লিডার নির্বাচন (Election)      │
│  ✅ ব্রোকার ফেইলওভার ম্যানেজ করা            │
│  ✅ টপিক তৈরি/ডিলিট অপারেশন               │
│  ✅ রেপ্লিকা অ্যাসাইনমেন্ট                  │
│  ✅ ISR লিস্ট আপডেট করা                     │
└─────────────────────────────────────────────┘
```

ধরুন Pathao-র রাইড সার্ভিসে Broker 2 হঠাৎ ক্র্যাশ করলো। Controller Broker সাথে সাথে বুঝতে পারবে
এবং Broker 2 তে যেসব পার্টিশন Leader ছিল, সেগুলোর জন্য নতুন Leader নির্বাচন করবে।

### 📌 ব্রোকার আবিষ্কার (Broker Discovery)

```
Producer/Consumer                     Kafka Cluster
       │                       ┌──────────────────────┐
       │  1. Connect to any    │                      │
       ├──────────────────────►│  Broker 1             │
       │     broker            │  (bootstrap server)   │
       │                       │                      │
       │  2. Get metadata      │                      │
       │◄──────────────────────┤  "Broker 2 has P0    │
       │  (all brokers +       │   Broker 3 has P1"   │
       │   partition leaders)  │                      │
       │                       └──────────────────────┘
       │
       │  3. Connect directly
       ├──────────────────────► Broker 2 (for P0)
       ├──────────────────────► Broker 3 (for P1)
```

ক্লায়েন্ট যেকোনো একটি ব্রোকারের সাথে কানেক্ট করলেই সম্পূর্ণ ক্লাস্টারের মেটাডাটা পেয়ে যায়।
এরপর সরাসরি সঠিক ব্রোকারের সাথে কথা বলে — কোনো প্রক্সি লাগে না!

## 🔄 ZooKeeper vs KRaft

### 📌 ZooKeeper কী করতো?

দীর্ঘদিন ধরে Kafka চালাতে একটি আলাদা সিস্টেম **ZooKeeper** লাগতো। এটি ক্লাস্টারের মেটাডাটা
ম্যানেজ করতো — কোন ব্রোকার জীবিত, কোন পার্টিশনের লিডার কে, কনজিউমার গ্রুপ ইনফো ইত্যাদি।

```
ZooKeeper-ভিত্তিক আর্কিটেকচার (পুরাতন):

  ┌─────────────────────────────────────────────────────┐
  │              ZooKeeper Ensemble                       │
  │   ┌─────────┐   ┌─────────┐   ┌─────────┐          │
  │   │  ZK 1   │   │  ZK 2   │   │  ZK 3   │          │
  │   │(Leader) │   │(Follower)│  │(Follower)│          │
  │   └────┬────┘   └────┬────┘   └────┬────┘          │
  │        └──────────────┼──────────────┘               │
  └───────────────────────┼─────────────────────────────┘
                          │
              Metadata Management
              (Controller Election,
               Topic Config, ACLs)
                          │
  ┌───────────────────────┼─────────────────────────────┐
  │                 Kafka Cluster                        │
  │   ┌─────────┐   ┌────┴────┐   ┌─────────┐         │
  │   │Broker 1 │   │Broker 2 │   │Broker 3 │         │
  │   │         │   │(Ctrl)   │   │         │         │
  │   └─────────┘   └─────────┘   └─────────┘         │
  └─────────────────────────────────────────────────────┘
```

### 📌 ZooKeeper-এর সমস্যা

ধরুন Daraz-এর ইভেন্ট স্ট্রিমিং প্ল্যাটফর্ম ম্যানেজ করছেন:

| সমস্যা | ব্যাখ্যা |
|---------|----------|
| **অতিরিক্ত জটিলতা** | Kafka ছাড়াও আলাদা ZooKeeper ক্লাস্টার ম্যানেজ করতে হতো |
| **স্কেলিং সীমাবদ্ধতা** | ZooKeeper ২০০,০০০+ পার্টিশনে ধীর হয়ে যেত |
| **ধীর রিকভারি** | Controller failover-এ মিনিট খরচ হতো |
| **ডুয়াল সিস্টেম** | দুটো আলাদা সিস্টেমের মনিটরিং, আপগ্রেড, সিকিউরিটি |

### 📌 KRaft মোড (KIP-500)

Kafka 3.3+ থেকে **KRaft** (Kafka Raft) এসেছে — ZooKeeper আর লাগে না!
এখন Kafka নিজেই নিজের মেটাডাটা ম্যানেজ করে Raft consensus protocol ব্যবহার করে।

```
KRaft-ভিত্তিক আর্কিটেকচার (নতুন):

  ┌─────────────────────────────────────────────────────┐
  │                   Kafka Cluster                      │
  │                                                     │
  │   ┌─────────────┐  ┌─────────────┐  ┌────────────┐│
  │   │  Broker 1   │  │  Broker 2   │  │  Broker 3  ││
  │   │ + Controller │  │ + Controller │  │+ Controller││
  │   │  (Active)   │  │  (Standby)  │  │ (Standby)  ││
  │   └──────┬──────┘  └──────┬──────┘  └─────┬──────┘│
  │          │                │                │       │
  │          └────────────────┼────────────────┘       │
  │                           │                        │
  │                  Raft Consensus                     │
  │              (Internal Metadata Log)                │
  │           ❌ ZooKeeper লাগবে না!                    │
  └─────────────────────────────────────────────────────┘
```

### 📌 তুলনা টেবিল

| বৈশিষ্ট্য | ZooKeeper মোড | KRaft মোড |
|-----------|--------------|-----------|
| **বহিরাগত নির্ভরতা** | ZooKeeper ক্লাস্টার দরকার | ❌ কিছু লাগে না |
| **পার্টিশন সীমা** | ~২০০K পার্টিশন | লক্ষাধিক পার্টিশন |
| **Controller Failover** | কয়েক মিনিট | কয়েক সেকেন্ড |
| **ডিপ্লয়মেন্ট** | জটিল (দুই সিস্টেম) | সরল (এক সিস্টেম) |
| **মেটাডাটা প্রচার** | ZK watch + RPC | Event-driven log |
| **Kafka সংস্করণ** | 0.x - 3.x | 3.3+ (production-ready 3.6+) |

### 📌 মাইগ্রেশন পাথ

```
ZooKeeper থেকে KRaft-এ মাইগ্রেশন:

  Step 1              Step 2              Step 3
  ┌──────────┐       ┌──────────┐       ┌──────────┐
  │ ZK মোডে  │       │ ডুয়াল    │       │ KRaft    │
  │ চালু আছে │──────►│ মোডে চলছে│──────►│ মোডে     │
  │          │       │ (ZK+KRaft)│       │ সম্পূর্ণ │
  └──────────┘       └──────────┘       └──────────┘
       │                  │                   │
   পুরাতন Kafka      মেটাডাটা মাইগ্রেট     ZooKeeper
   + ZooKeeper        হচ্ছে                 বন্ধ করুন
```

> 💡 **বাস্তব পরামর্শ:** নতুন প্রজেক্ট (যেমন Nagad-এর নতুন মাইক্রোসার্ভিস) সরাসরি KRaft মোডে শুরু করুন।

## 📦 টপিক পার্টিশনিং কৌশল

### 📌 পার্টিশন ডিস্ট্রিবিউশন

Kafka টপিকের পার্টিশনগুলো বিভিন্ন ব্রোকারে ছড়িয়ে দেয়। এতে লোড সমানভাবে ভাগ হয়।

```
Topic: bkash-transactions (6 Partitions, 3 Brokers)

  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │  Broker 1   │    │  Broker 2   │    │  Broker 3   │
  │             │    │             │    │             │
  │  ┌───────┐  │    │  ┌───────┐  │    │  ┌───────┐  │
  │  │  P0   │  │    │  │  P1   │  │    │  │  P2   │  │
  │  └───────┘  │    │  └───────┘  │    │  └───────┘  │
  │  ┌───────┐  │    │  ┌───────┐  │    │  ┌───────┐  │
  │  │  P3   │  │    │  │  P4   │  │    │  │  P5   │  │
  │  └───────┘  │    │  └───────┘  │    │  └───────┘  │
  │             │    │             │    │             │
  └─────────────┘    └─────────────┘    └─────────────┘

  প্রতি ব্রোকারে ২টি করে পার্টিশন = সমান লোড ডিস্ট্রিবিউশন
```

### 📌 পার্টিশন লিডার নির্বাচন

প্রতিটি পার্টিশনের একটি **Leader** থাকে। সব রিড/রাইট Leader-এর কাছে যায়।

```
পার্টিশন P0-এর Leader Election:

  সময় T1 (স্বাভাবিক):
  ┌────────┐   ┌────────┐   ┌────────┐
  │Broker 1│   │Broker 2│   │Broker 3│
  │ P0 (L) │   │ P0 (F) │   │ P0 (F) │   ← Broker 1 হলো Leader
  └────────┘   └────────┘   └────────┘

  সময় T2 (Broker 1 ক্র্যাশ!):
  ┌────────┐   ┌────────┐   ┌────────┐
  │Broker 1│   │Broker 2│   │Broker 3│
  │  ❌💀  │   │ P0 (F) │   │ P0 (F) │   ← Leader নেই!
  └────────┘   └────────┘   └────────┘

  সময় T3 (Controller নতুন Leader নির্বাচন করে):
  ┌────────┐   ┌────────┐   ┌────────┐
  │Broker 1│   │Broker 2│   │Broker 3│
  │  ❌💀  │   │ P0 (L) │   │ P0 (F) │   ← Broker 2 নতুন Leader!
  └────────┘   └────────┘   └────────┘
```

### 📌 পার্টিশন সংখ্যা নির্ধারণ

পার্টিশন সংখ্যা খুবই গুরুত্বপূর্ণ সিদ্ধান্ত। উদাহরণ — Grameenphone-র ইভেন্ট সিস্টেম:

| ফ্যাক্টর | কম পার্টিশন (3-5) | বেশি পার্টিশন (50+) |
|----------|-------------------|---------------------|
| **Throughput** | কম | বেশি |
| **Ordering** | সহজে গ্যারান্টি | শুধু পার্টিশনের মধ্যে |
| **Consumer** | কম consumer লাগে | বেশি consumer দিতে পারবেন |
| **মেমোরি** | কম | বেশি |
| **Failover সময়** | দ্রুত | ধীর |
| **End-to-end Latency** | কম | বেশি হতে পারে |

> 🎯 **সূত্র:** `পার্টিশন সংখ্যা ≥ সর্বোচ্চ consumer সংখ্যা`

### 📌 Key-ভিত্তিক পার্টিশনিং

Key দিলে Kafka নির্দিষ্ট পার্টিশনে মেসেজ পাঠায়। একই key = একই পার্টিশন = ক্রম বজায়।

```
bKash ট্রানজ্যাকশন — Key: user_phone_number

  Key: 01711xxxxxx ──hash()──► P0  (এই ইউজারের সব ট্রানজ্যাকশন P0-তে)
  Key: 01812xxxxxx ──hash()──► P2  (এই ইউজারের সব ট্রানজ্যাকশন P2-তে)
  Key: 01911xxxxxx ──hash()──► P1  (এই ইউজারের সব ট্রানজ্যাকশন P1-তে)
  Key: 01711xxxxxx ──hash()──► P0  (আবারও P0! কারণ একই key)

  ┌─────────────────────────────────────────────────────┐
  │  hash(key) % num_partitions = target partition       │
  │  hash("01711xxxxxx") % 6 = 0  → P0                  │
  │  hash("01812xxxxxx") % 6 = 2  → P2                  │
  └─────────────────────────────────────────────────────┘
```

## 🔄 রেপ্লিকেশন (Replication)

### 📌 কেন রেপ্লিকেশন দরকার?

Kafka-তে ডাটা হারানো যাবে না — বিশেষ করে Nagad-এর মতো ফিনান্সিয়াল সিস্টেমে। রেপ্লিকেশন
নিশ্চিত করে যে একটি ব্রোকার মারা গেলেও ডাটা অন্য ব্রোকারে নিরাপদে আছে।

### 📌 Leader ও Follower রেপ্লিকা

```
Topic: orders (Replication Factor = 3)

  ┌──────────────┬──────────────┬──────────────┐
  │   Broker 1   │   Broker 2   │   Broker 3   │
  ├──────────────┼──────────────┼──────────────┤
  │   P0 (L) ★  │   P0 (F)     │   P0 (F)     │
  │   P1 (F)     │   P1 (L) ★  │   P1 (F)     │
  │   P2 (F)     │   P2 (F)     │   P2 (L) ★  │
  └──────────────┴──────────────┴──────────────┘

  L = Leader (★ রিড/রাইট এখানে হয়)
  F = Follower (Leader থেকে ডাটা কপি করে)
```

### 📌 ISR (In-Sync Replicas) কনসেপ্ট

ISR হলো সেই রেপ্লিকাগুলোর লিস্ট যারা Leader-এর সাথে **আপ-টু-ডেট** আছে।

```
স্বাভাবিক অবস্থা — সব রেপ্লিকা ISR-এ:

  Leader (Broker 1)     Follower (Broker 2)   Follower (Broker 3)
  ┌──────────────┐     ┌──────────────┐      ┌──────────────┐
  │ Offset: 100  │────►│ Offset: 100  │      │ Offset: 100  │
  │              │     │              │◄─────│              │
  │  [msg1]      │     │  [msg1]      │      │  [msg1]      │
  │  [msg2]      │     │  [msg2]      │      │  [msg2]      │
  │  [msg3]      │     │  [msg3]      │      │  [msg3]      │
  └──────────────┘     └──────────────┘      └──────────────┘
         ISR = {Broker1, Broker2, Broker3}  ✅ সব সিঙ্ক!

  যখন Broker 3 পিছিয়ে পড়ে:

  Leader (Broker 1)     Follower (Broker 2)   Follower (Broker 3)
  ┌──────────────┐     ┌──────────────┐      ┌──────────────┐
  │ Offset: 150  │────►│ Offset: 148  │      │ Offset: 80   │
  │              │     │   (কাছাকাছি) │      │   (অনেক দূরে!)│
  └──────────────┘     └──────────────┘      └──────────────┘
         ISR = {Broker1, Broker2}  ⚠️ Broker 3 ISR থেকে বাদ!
```

### 📌 ফেইলওভার (Leader Failure)

```
ধাপ ১: Leader ব্রোকার ক্র্যাশ করলো

  ┌────────────┐    ┌────────────┐    ┌────────────┐
  │  Broker 1  │    │  Broker 2  │    │  Broker 3  │
  │  P0 (L) ★  │    │  P0 (F)    │    │  P0 (F)    │
  │    ❌💥     │    │ ISR ✅     │    │ ISR ✅     │
  └────────────┘    └────────────┘    └────────────┘

ধাপ ২: Controller ISR থেকে নতুন Leader বাছাই করে

  ┌────────────┐    ┌────────────┐    ┌────────────┐
  │  Broker 1  │    │  Broker 2  │    │  Broker 3  │
  │    ❌💀     │    │  P0 (L) ★  │    │  P0 (F)    │
  │   (মৃত)    │    │ নতুন Leader│    │            │
  └────────────┘    └────────────┘    └────────────┘

ধাপ ৩: Broker 1 ফিরে আসলে Follower হিসেবে যোগ দেয়

  ┌────────────┐    ┌────────────┐    ┌────────────┐
  │  Broker 1  │    │  Broker 2  │    │  Broker 3  │
  │  P0 (F)    │    │  P0 (L) ★  │    │  P0 (F)    │
  │ (ফিরে এসেছে)│   │  (Leader)  │    │            │
  └────────────┘    └────────────┘    └────────────┘
```

### 📌 min.insync.replicas

এটি নির্ধারণ করে কমপক্ষে কতগুলো রেপ্লিকা সিঙ্কে থাকতে **হবে** একটি write সফল হতে।

```
acks=all + min.insync.replicas=2 (Daraz অর্ডার সিস্টেম):

  Producer ──── Write Request ────► Leader (Broker 1)
                                        │
                                        ├──► Follower (Broker 2) ✅ ACK
                                        ├──► Follower (Broker 3) ✅ ACK
                                        │
                                   ISR = 3, min.insync = 2
                                   ✅ Write সফল!

  যদি ২টি follower মারা যায়:

  Producer ──── Write Request ────► Leader (Broker 1)
                                        │
                                        ├──► Follower (Broker 2) ❌ Dead
                                        ├──► Follower (Broker 3) ❌ Dead
                                        │
                                   ISR = 1, min.insync = 2
                                   ❌ NotEnoughReplicasException!
                                   (ডাটা সেফটির জন্য write প্রত্যাখ্যান)
```

### 📌 Replication Factor সুপারিশ

| পরিস্থিতি | Replication Factor | উদাহরণ |
|-----------|-------------------|---------|
| **Development** | 1 | লোকাল ডেভ মেশিন |
| **Non-critical** | 2 | লগ/অ্যানালিটিক্স |
| **Production** | 3 | bKash ট্রানজ্যাকশন, Pathao রাইড |
| **Mission-critical Finance** | 3 (min.insync=2) | Nagad পেমেন্ট লেজার |

## 📝 লগ সেগমেন্ট ও স্টোরেজ

### 📌 Kafka কিভাবে ডিস্কে ডাটা রাখে?

Kafka মেসেজগুলো **append-only log** হিসেবে ডিস্কে রাখে। প্রতিটি পার্টিশন একটি ডিরেক্টরি,
ভেতরে থাকে **log segment** ফাইল।

```
ডিস্কে ফাইল স্ট্রাকচার:

  /kafka-data/
  └── bkash-transactions-0/          ← Topic: bkash-transactions, Partition 0
      │
      ├── 00000000000000000000.log    ← প্রথম segment (offset 0 থেকে)
      ├── 00000000000000000000.index  ← offset → position ম্যাপিং
      ├── 00000000000000000000.timeindex ← timestamp → offset ম্যাপিং
      │
      ├── 00000000000000054321.log    ← দ্বিতীয় segment (offset 54321 থেকে)
      ├── 00000000000000054321.index
      ├── 00000000000000054321.timeindex
      │
      └── 00000000000000112233.log    ← তৃতীয় segment (Active — এখানে write হচ্ছে)
          00000000000000112233.index
          00000000000000112233.timeindex
```

### 📌 লগ সেগমেন্ট বিস্তারিত

```
একটি .log ফাইলের ভেতরের গঠন:

  ┌──────────────────────────────────────────────────────┐
  │              Log Segment File (.log)                   │
  │                                                       │
  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │
  │  │Record 0 │ │Record 1 │ │Record 2 │ │Record 3 │   │
  │  │         │ │         │ │         │ │         │   │
  │  │ offset  │ │ offset  │ │ offset  │ │ offset  │   │
  │  │ size    │ │ size    │ │ size    │ │ size    │   │
  │  │ key     │ │ key     │ │ key     │ │ key     │   │
  │  │ value   │ │ value   │ │ value   │ │ value   │   │
  │  │ headers │ │ headers │ │ headers │ │ headers │   │
  │  │timestamp│ │timestamp│ │timestamp│ │timestamp│   │
  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘   │
  │                                                       │
  │  ← Append-only: নতুন রেকর্ড সবসময় শেষে যোগ হয় →     │

  ┌──────────────────────────────┐

```

### 📌 সেগমেন্ট রোটেশন

```
Active Segment পূর্ণ হলে নতুন segment তৈরি হয়:

  সময় T1:                     সময় T2 (segment পূর্ণ):
  ┌─────────────────┐          ┌─────────────────┐
  │ Segment 0       │          │ Segment 0       │  ← Sealed (আর write নেই)
  │ [Active]        │   ──►    │ [Immutable]     │
  │ offset 0-99999  │          │ offset 0-99999  │
  └─────────────────┘          └─────────────────┘
                               ┌─────────────────┐
                               │ Segment 1       │  ← নতুন Active
                               │ [Active]        │
                               │ offset 100000+  │
                               └─────────────────┘

  segment.bytes = 1GB (ডিফল্ট)

```

### 📌 লগ কম্প্যাকশন (Log Compaction)

লগ কম্প্যাকশন শুধু প্রতিটি key-র **সর্বশেষ** মান রাখে। এটি KTable-স্টাইল টপিকের জন্য দারুণ।

```
কম্প্যাকশনের আগে:                    কম্প্যাকশনের পরে:

  Offset | Key    | Value             Offset | Key    | Value
  ───────┼────────┼──────             ───────┼────────┼──────
  0      | user-1 | Rahim              3     | user-1 | Rahim_v2
  1      | user-2 | Karim              4     | user-2 | Karim_v2
  2      | user-3 | Salma              2     | user-3 | Salma
  3      | user-1 | Rahim_v2
  4      | user-2 | Karim_v2

  পুরাতন ডুপ্লিকেট মুছে ফেলা হয়েছে — শুধু সর্বশেষ মান আছে!
```

## ⚡ কিভাবে Kafka High Throughput অর্জন করে

### 📌 Kafka এত দ্রুত কেন?

Pathao-র সিস্টেমে প্রতি সেকেন্ডে লক্ষ লক্ষ GPS ইভেন্ট প্রসেস হয়। Kafka এটি সম্ভব করে
কিছু চমৎকার ইঞ্জিনিয়ারিং ট্রিকের মাধ্যমে:

### 📌 Zero-Copy Transfer

সাধারণ ডাটা ট্রান্সফারে ডাটা ৪ বার কপি হয়। Kafka-র `sendfile()` syscall এটি ২ বারে নামিয়ে আনে।

```
❌ Traditional Transfer (৪ বার কপি):

  Disk ──①──► Kernel Buffer ──②──► User Space (App)
                                       │
  NIC ◄──④── Kernel Buffer ◄──③──┘ (Socket Buffer)

  মোট: ৪টি কপি + ২টি context switch


✅ Zero-Copy Transfer (sendfile):

  Disk ──①──► Kernel Buffer ──②──► NIC (সরাসরি!)

  মোট: ২টি কপি + ০ context switch
  🚀 ৬০-৭০% কম CPU ব্যবহার!


  তুলনা:
  ┌────────────────────┬──────────────┬──────────────┐
  │                    │ Traditional  │  Zero-Copy   │
  ├────────────────────┼──────────────┼──────────────┤
  │ Data Copies        │      4       │      2       │
  │ Context Switches   │      4       │      2       │
  │ CPU Usage          │    High      │    Low       │
  │ Throughput         │   ~100MB/s   │   ~600MB/s   │
  └────────────────────┴──────────────┴──────────────┘
```

### 📌 Sequential I/O

```
Random I/O (Traditional DB):          Sequential I/O (Kafka):

  Disk Head Movement:                  Disk Head Movement:

    ┌─┐   ┌─┐                          ┌─┬─┬─┬─┬─┬─┬─┐
    │ │   │ │   ┌─┐                     │ │ │ │ │ │ │ │
    │ │   │ │   │ │  ┌─┐               │ │ │ │ │ │ │ │
    └─┘   └─┘   └─┘  └─┘               └─┴─┴─┴─┴─┴─┴─┘
    ↑  ↗    ↑  ↗                        ──────────────►
    এদিক ওদিক লাফাচ্ছে                 একটানা সামনে যাচ্ছে

  Speed: ~100 IOPS                     Speed: ~600 MB/s
  (HDD-তে খুবই ধীর)                    (SSD-র চেয়েও দ্রুত হতে পারে!)
```

### 📌 OS Page Cache

Kafka সরাসরি OS-এর page cache ব্যবহার করে। আলাদা কোনো in-process cache নেই।

```
  Producer                                   Consumer
     │                                          ▲
     ▼                                          │
  ┌──────────────────────────────────────────────┤
  │           Kafka Broker (JVM)                 │
  │    (শুধু ডাটা pass-through করে,              │
  │     নিজে cache করে না)                      │
  └──────────┬───────────────────────────────────┘
             │
  ┌──────────▼───────────────────────────────────┐
  │           OS Page Cache (RAM)                 │
  │                                              │
  │  সাম্প্রতিক ডাটা RAM-এ থাকে!                 │
  │  Consumer পড়ার সময় disk-এ যেতে হয় না       │
  │  (যদি ডাটা "তাজা" হয়)                       │
  └──────────┬───────────────────────────────────┘
             │
  ┌──────────▼───────────────────────────────────┐
  │              Disk (SSD/HDD)                   │
  │         পুরাতন ডাটা এখানে                     │
  └──────────────────────────────────────────────┘
```

### 📌 Batching ও Compression

```
❌ ব্যাচিং ছাড়া:                    ✅ ব্যাচিং সহ:

  msg1 ──► Network ──► Broker        [msg1, msg2, msg3, ... msg100]
  msg2 ──► Network ──► Broker              │
  msg3 ──► Network ──► Broker              ▼
  ...                                  Network (১ বার)
  msg100 ──► Network ──► Broker            │
                                           ▼
  ১০০টি network round-trip!            Broker (১ বার!)

  linger.ms=5        ← ৫ms অপেক্ষা করে batch তৈরি করে
  batch.size=16384   ← 16KB পর্যন্ত batch করে
  compression=snappy ← কমপ্রেস করে সাইজ ৬০-৮০% কমায়
```

## 🗑️ ডাটা রিটেনশন পলিসি

### 📌 সময়-ভিত্তিক রিটেনশন

Kafka ডাটা চিরকাল রাখে না — নির্দিষ্ট সময় পর স্বয়ংক্রিয়ভাবে মুছে ফেলে।

```
সময়-ভিত্তিক রিটেনশন (log.retention.hours):

  ┌──────────────────────────────────────────────────────────┐
  │                    Timeline                               │
  │                                                          │
  │  7 days ago    5 days ago    3 days ago    Now            │
  │  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
  │  │ Segment │  │ Segment │  │ Segment │  │ Segment │    │
  │  │    1    │  │    2    │  │    3    │  │    4    │    │
  │  │  🗑️     │  │         │  │         │  │ [Active]│    │
  │  │ DELETE  │  │         │  │         │  │         │    │
  │  └─────────┘  └─────────┘  └─────────┘  └─────────┘    │
  │                                                          │
  │  ←── log.retention.hours=168 (7 দিন) ──►                 │
  │      7 দিনের পুরনো segment মুছে ফেলা হবে                │
  └──────────────────────────────────────────────────────────┘
```

### 📌 সাইজ-ভিত্তিক রিটেনশন

```
সাইজ-ভিত্তিক রিটেনশন (log.retention.bytes):

  log.retention.bytes = 10GB (প্রতি পার্টিশনে)

  পার্টিশনের মোট সাইজ > 10GB হলে:
  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
  │ 3GB     │ │ 3GB     │ │ 3GB     │ │ 2GB     │ = 11GB
  │ 🗑️      │ │         │ │         │ │ [Active]│
  │ (মুছবে) │ │         │ │         │ │         │
  └─────────┘ └─────────┘ └─────────┘ └─────────┘
  ← পুরাতন segment মুছে 10GB-র নিচে আনবে
```

### 📌 কনফিগারেশন উদাহরণ

```properties
# Grameenphone ইভেন্ট সিস্টেমের জন্য উদাহরণ কনফিগারেশন:

# ===== সময়-ভিত্তিক রিটেনশন =====
log.retention.hours=168          # 7 দিন (ডিফল্ট)
# log.retention.minutes=10080   # মিনিটেও দেওয়া যায়
# log.retention.ms=604800000    # মিলিসেকেন্ডেও দেওয়া যায়

# ===== সাইজ-ভিত্তিক রিটেনশন =====
log.retention.bytes=10737418240  # 10 GB per partition
# log.retention.bytes=-1        # সীমাহীন (ডিফল্ট)

# ===== সেগমেন্ট কনফিগ =====
log.segment.bytes=1073741824    # 1 GB per segment
log.segment.ms=604800000        # 7 days per segment

# ===== লগ কম্প্যাকশন =====
log.cleanup.policy=compact       # শুধু কম্প্যাকশন
# log.cleanup.policy=delete      # শুধু ডিলিট (ডিফল্ট)
# log.cleanup.policy=compact,delete  # দুটোই

# ===== কম্প্যাকশন সেটিংস =====
log.cleaner.min.cleanable.ratio=0.5
log.cleaner.min.compaction.lag.ms=0
```

### 📌 Compact vs Delete পলিসি

| পলিসি | ব্যবহার | উদাহরণ |
|-------|---------|---------|
| **delete** | সময়/সাইজ অনুসারে পুরাতন ডাটা মুছে ফেলে | Pathao রাইড GPS ইভেন্ট |
| **compact** | প্রতি key-র শেষ মান রাখে | ইউজার প্রোফাইল আপডেট |
| **compact,delete** | কম্প্যাকশন + সময়ের পর ডিলিট | সেশন ডাটা (শেষ মান রাখো, ৩০ দিন পর মুছো) |

### 📌 পার্টিশন সংখ্যা কিভাবে ঠিক করবেন?

```
সূত্র:
  Target Throughput = T MB/s
  প্রতি পার্টিশনের throughput = P MB/s (সাধারণত 10-50 MB/s)

  পার্টিশন সংখ্যা = ceil(T / P)

  উদাহরণ — Daraz-এর 11.11 সেলে:
  ┌─────────────────────────────────────────────┐
  │ Target throughput: 200 MB/s                  │
  │ প্রতি পার্টিশন: ~20 MB/s                     │
  │ পার্টিশন সংখ্যা = 200 / 20 = 10             │
  │                                              │
  │ কিন্তু consumer parallelism চাই 15           │
  │ সুতরাং পার্টিশন = max(10, 15) = 15           │
  │                                              │
  │ ভবিষ্যৎ growth (2x): 15 × 2 = 30            │
  │ ✅ চূড়ান্ত সিদ্ধান্ত: 30 পার্টিশন             │
  └─────────────────────────────────────────────┘
```

> ⚠️ **সতর্কতা:** পার্টিশন সংখ্যা বাড়ানো সহজ কিন্তু **কমানো যায় না**। তাই চিন্তাভাবনা করে ঠিক করুন।
