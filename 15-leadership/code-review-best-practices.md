### কোড রিভিউয়ের মূল সুবিধাগুলো

**১. বাগ প্রতিরোধ (Bug Prevention):**
প্রোডাকশনে যাওয়ার আগেই সমস্যা ধরা পড়ে। Pathao-র রাইড ক্যালকুলেশনে ভুল থাকলে
লক্ষ লক্ষ ইউজার ভুল ভাড়া দেখবে — রিভিউতে ধরা পড়লে এই বিপদ এড়ানো যায়।

**২. জ্ঞান শেয়ারিং (Knowledge Sharing):**
জুনিয়র ডেভেলপাররা সিনিয়রদের কোড দেখে শেখে, সিনিয়ররা নতুন আইডিয়া পায়।
Grameenphone-র টিমে একজন কোড লিখলে পুরো টিম সেই কোড সম্পর্কে জানে।

**৩. কোড কোয়ালিটি (Code Quality):**
কনসিসটেন্ট কোডিং স্ট্যান্ডার্ড বজায় থাকে। Daraz-এর মতো বড় প্ল্যাটফর্মে ১০০+ ডেভেলপার
একই স্টাইলে কোড লেখে — কারণ রিভিউ প্রক্রিয়া চালু আছে।

**৪. সিকিউরিটি (Security):**
SQL injection, XSS, CSRF এর মতো ভালনারেবিলিটি রিভিউতে ধরা পড়ে। Nagad-এর মতো
ফিনটেক কোম্পানিতে এটি অত্যন্ত জরুরি।

### ১. সঠিকতা (Correctness)

কোড কি আসলেই যা করার কথা তাই করছে? লজিক কি সঠিক?

#### ❌ খারাপ কোড — bKash ব্যালেন্স চেক

```php
<?php
// ❌ খারাপ: Race condition আছে — দুটো রিকোয়েস্ট একসাথে আসলে
// ব্যালেন্সের চেয়ে বেশি টাকা ট্রান্সফার হতে পারে
function transferMoney($fromUser, $toUser, $amount) {
    $balance = DB::table('wallets')
        ->where('user_id', $fromUser)
        ->value('balance');

    if ($balance >= $amount) {
        // এই দুই লাইনের মাঝে আরেকটা ট্রান্সফার হতে পারে!
        DB::table('wallets')
            ->where('user_id', $fromUser)
            ->decrement('balance', $amount);

        DB::table('wallets')
            ->where('user_id', $toUser)
            ->increment('balance', $amount);
    }
}
```

#### ✅ ভালো কোড — ট্রানজ্যাকশন ও লকিং সহ

```php
<?php
// ✅ ভালো: ট্রানজ্যাকশন ও pessimistic locking ব্যবহার করা হয়েছে
function transferMoney($fromUser, $toUser, $amount) {
    return DB::transaction(function () use ($fromUser, $toUser, $amount) {
        $wallet = DB::table('wallets')
            ->where('user_id', $fromUser)
            ->lockForUpdate()  // অন্য কেউ এই রো পড়তে/লিখতে পারবে না
            ->first();

        if ($wallet->balance < $amount) {
            throw new InsufficientBalanceException(
                "অপর্যাপ্ত ব্যালেন্স: প্রয়োজন ৳{$amount}, আছে ৳{$wallet->balance}"
            );
        }

        DB::table('wallets')
            ->where('user_id', $fromUser)
            ->decrement('balance', $amount);

        DB::table('wallets')
            ->where('user_id', $toUser)
            ->increment('balance', $amount);

        TransactionLog::create([
            'from_user' => $fromUser,
            'to_user'   => $toUser,
            'amount'    => $amount,
            'status'    => 'completed',
        ]);

        return true;
    });
}
```

### ৩. সিকিউরিটি (Security)

#### ❌ খারাপ: SQL Injection সম্ভব

```php
<?php
// ❌ বিপজ্জনক! ইউজার ইনপুট সরাসরি কোয়েরিতে বসানো হচ্ছে
function searchProducts($keyword) {
    $query = "SELECT * FROM products WHERE name LIKE '%{$keyword}%'";
    return DB::select($query);
}
// হ্যাকার ইনপুট দিতে পারে: ' OR 1=1; DROP TABLE products; --
```

#### ✅ ভালো: প্যারামিটারাইজড কোয়েরি

```php
<?php
// ✅ নিরাপদ: প্যারামিটার বাইন্ডিং ব্যবহার করা হয়েছে
function searchProducts($keyword) {
    return DB::table('products')
        ->where('name', 'LIKE', '%' . $keyword . '%')
        ->get();
    // Laravel নিজেই ইনপুট sanitize করবে

```

### ৪. রিডেবিলিটি (Readability)

#### ❌ খারাপ JavaScript — Pathao ভাড়া ক্যালকুলেশন

```javascript
// ❌ খারাপ: কী হচ্ছে বোঝা অসম্ভব
function c(d, t) {
  let r = d * 12;
  if (t === 1) r *= 1.5;
  else if (t === 2) r *= 2.0;
  if (d > 10) r += 50;
  return r > 25 ? r : 25;
}
```

#### ✅ ভালো JavaScript — স্পষ্ট নামকরণ ও স্ট্রাকচার

```javascript
// ✅ ভালো: পরিষ্কার, বোঝা যায়
const RATE_PER_KM = 12;
const MINIMUM_FARE = 25;
const LONG_DISTANCE_BONUS = 50;
const LONG_DISTANCE_THRESHOLD_KM = 10;

const VEHICLE_MULTIPLIERS = {
  bike: 1.0,
  cng: 1.5,
  car: 2.0,
};

function calculateRideFare(distanceKm, vehicleType) {
  let fare = distanceKm * RATE_PER_KM;

  const multiplier = VEHICLE_MULTIPLIERS[vehicleType] || 1.0;
  fare *= multiplier;

  if (distanceKm > LONG_DISTANCE_THRESHOLD_KM) {
    fare += LONG_DISTANCE_BONUS;
  }

  return Math.max(fare, MINIMUM_FARE);
}
```

### ফিডব্যাকের ভাষা

রিভিউ কমেন্টে prefix ব্যবহার করুন যাতে author বুঝতে পারে কোনটা কতটুকু গুরুত্বপূর্ণ:

| Prefix | অর্থ | উদাহরণ |
|--------|-------|---------|
| `[blocker]` | এটি ঠিক না করলে merge হবে না | `[blocker] SQL injection ভালনারেবিলিটি আছে` |
| `[suggestion]` | উন্নতির পরামর্শ, বাধ্যতামূলক নয় | `[suggestion] এখানে early return ব্যবহার করা যেতে পারে` |
| `[question]` | বুঝতে চাইছি, সমস্যা নাও হতে পারে | `[question] এই timeout value কেন ৩০ সেকেন্ড?` |
| `[nit]` | ছোট বিষয়, না করলেও চলে | `[nit] ভ্যারিয়েবলের নামটা আরেকটু descriptive হলে ভালো হতো` |
| `[praise]` | ভালো কাজের প্রশংসা | `[praise] চমৎকার error handling!` |

### কেন ছোট PR ভালো?

**Daraz-এর উদাহরণ:** ধরুন একজন ডেভেলপার পুরো প্রোডাক্ট ক্যাটালগ ফিচার একটা PR-এ
দিলেন — ৩০০০ লাইন! রিভিউয়ার ক্লান্ত হয়ে শুধু উপরের দিকটা দেখে approve করে দেবেন।
অথচ নিচের দিকে একটা critical bug থাকতে পারে।

বরং ৫টা ছোট PR-এ ভাগ করলে প্রতিটি ভালোভাবে রিভিউ হবে:

```
PR #1: প্রোডাক্ট মডেল ও মাইগ্রেশন        (~100 লাইন)
PR #2: প্রোডাক্ট CRUD API                  (~150 লাইন)
PR #3: প্রোডাক্ট সার্চ ও ফিল্টারিং         (~120 লাইন)
PR #4: প্রোডাক্ট ইমেজ আপলোড               (~80 লাইন)
PR #5: প্রোডাক্ট ক্যাটালগ ফ্রন্টএন্ড UI      (~200 লাইন)
```

### কখন বড় PR গ্রহণযোগ্য

```yaml
name: Code Quality Check

on:
  pull_request:
    branches: [main, develop]

jobs:
  sonarqube:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Run Tests with Coverage
        run: |
          composer install --no-interaction

      - name: SonarQube Scan
        uses: sonarsource/sonarqube-scan-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: Quality Gate Check

        timeout-minutes: 5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

## 📊 ভালো vs খারাপ রিভিউ কমেন্ট

রিভিউ কমেন্ট কীভাবে লিখবেন তা শিখুন এই উদাহরণগুলো দেখে। প্রতিটি জোড়ায়
একই সমস্যার জন্য খারাপ ও ভালো কমেন্ট দেখানো হয়েছে।

### জোড়া ১: সিকিউরিটি সমস্যা

```
❌ খারাপ কমেন্ট:
"এটা ভুল।"

✅ ভালো কমেন্ট:
"[blocker] এই ফাংশনে ইউজার ইনপুট সরাসরি SQL কোয়েরিতে
ব্যবহার হচ্ছে, যা SQL injection-এর ঝুঁকি তৈরি করে।
প্যারামিটার বাইন্ডিং ব্যবহার করলে সমস্যাটি সমাধান হবে:

$users = DB::select('SELECT * FROM users WHERE id = ?', [$id]);

OWASP SQL Injection: https://owasp.org/www-community/attacks/SQL_Injection"
```

### জোড়া ২: পারফরম্যান্স

```
❌ খারাপ কমেন্ট:
"এটা স্লো হবে।"

✅ ভালো কমেন্ট:
"[suggestion] এখানে লুপের ভেতরে প্রতিবার DB কোয়েরি চলছে।
১০০০ প্রোডাক্ট থাকলে ১০০০টি কোয়েরি হবে (N+1 সমস্যা)।

Eager loading ব্যবহার করলে মাত্র ২টি কোয়েরিতে কাজ হবে:
Product::with('category')->get();

Daraz-এ এই ধরনের সমস্যা হোমপেজ লোড ৩ সেকেন্ড থেকে ০.৫
সেকেন্ডে নামিয়ে এনেছিল।"
```

### জোড়া ৩: রিডেবিলিটি

```
❌ খারাপ কমেন্ট:
```

### জোড়া ৪: এরর হ্যান্ডলিং

```
❌ খারাপ কমেন্ট:

✅ ভালো কমেন্ট:
```

## 🎯 কখন ব্যবহার করবেন / করবেন না

### ✅ কোড রিভিউ করবেন যখন:

- **সব প্রোডাকশন কোডে** — কোনো ব্যতিক্রম নেই
- **কনফিগারেশন ফাইল পরিবর্তনে** — `.env`, `config/`, infrastructure
- **ডেটাবেজ মাইগ্রেশনে** — স্কিমা পরিবর্তন ভুল হলে ডেটা হারাতে পারে
- **সিকিউরিটি সম্পর্কিত কোডে** — অথেনটিকেশন, অথরাইজেশন, পেমেন্ট
- **শেয়ার্ড লাইব্রেরি/প্যাকেজে** — অন্যরাও ব্যবহার করে

- **ব্যক্তিগত নোটবুক/স্ক্র্যাচ ফাইল** — পরীক্ষামূলক কোড

```php
<?php
// app/Http/Controllers/RechargeController.php

class RechargeController extends Controller
{
    public function recharge(Request $request)
    {
        $phone = $request->phone;
        $amount = $request->amount;
        $operator = $request->operator;

        // ব্যালেন্স চেক
        $user = User::find($request->user_id);
        if ($user->balance < $amount) {
            return response()->json(['error' => 'Low balance']);
        }

        // রিচার্জ করা
        $user->balance -= $amount;
        $user->save();

        // অপারেটর API কল
        $result = Http::post("https://api.{$operator}.com/recharge", [
            'phone' => $phone,
            'amount' => $amount,
        ]);

        return response()->json(['success' => true]);
    }
}
```
