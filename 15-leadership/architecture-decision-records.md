## 📖 ১. ADR কী? (What are Architecture Decision Records?)

ADR হলো একটি **সংক্ষিপ্ত ডকুমেন্ট** যা কোনো গুরুত্বপূর্ণ আর্কিটেকচার ডিসিশনের
**প্রেক্ষাপট (Context)**, **সিদ্ধান্ত (Decision)** এবং **পরিণতি (Consequences)** রেকর্ড করে।

ধরুন আপনি bKash-এর ব্যাকএন্ড টিমে কাজ করছেন। একদিন সিদ্ধান্ত নিলেন PostgreSQL
ব্যবহার করবেন MySQL-এর বদলে। ৬ মাস পরে নতুন একজন ডেভেলপার জিজ্ঞেস করলো —
"ভাই, MySQL না কেন?" — ADR-তে এই প্রশ্নের উত্তর আগে থেকেই লেখা থাকে।

```
+---------------------------------------------------------------+
|                    ADR কী করে?                                  |
+---------------------------------------------------------------+
|                                                               |
|   ১. কী সিদ্ধান্ত নেওয়া হয়েছে?     (WHAT was decided?)      |
|   ২. কেন এই সিদ্ধান্ত?              (WHY this decision?)     |
|   ৩. কী কী বিকল্প ছিল?             (WHAT were the options?)  |
|   ৪. কী প্রভাব পড়বে?               (WHAT are the impacts?)   |
|                                                               |
+---------------------------------------------------------------+
```

### 🏠 বাস্তব উদাহরণ

বাংলাদেশের বড় টেক কোম্পানিগুলোতে ADR কিভাবে কাজ করে:

```
+------------------+----------------------------------------+
| কোম্পানি         | ADR-এর বিষয়                             |
+------------------+----------------------------------------+
| bKash            | PostgreSQL vs MySQL সিদ্ধান্ত           |
| Pathao           | Kafka vs RabbitMQ মেসেজ কিউ সিলেকশন    |
| Daraz            | REST vs GraphQL API ডিজাইন              |
| Nagad            | Microservices vs Monolith আর্কিটেকচার   |
| Grameenphone     | On-premise vs Cloud মাইগ্রেশন            |
+------------------+----------------------------------------+
```

### সমস্যা: "কেন এটা এভাবে করা হয়েছে?"

```
    ৬ মাস আগে                          আজকে
    +-----------+                    +-----------+
    |  সিনিয়র   |                    |  নতুন      |
    | ডেভেলপার  |   ---সময়--->      | ডেভেলপার   |

    |  নিয়েছে   |                    |            |
    +-----------+                    +-----------+
         |                                |
         |  ADR ছাড়া:                     |
         |  "আমি ভুলে গেছি"               |

         |  ADR থাকলে:                     |
```

## 📋 ৩. ADR টেমপ্লেট (Full ADR Template)

### সম্পূর্ণ টেমপ্লেট

```markdown
# ADR-[নম্বর]: [সংক্ষিপ্ত শিরোনাম]

## Status
[Proposed | Accepted | Deprecated | Superseded by ADR-XXX]

## Date
[YYYY-MM-DD]

## Context
[সমস্যা বা পরিস্থিতি কী ছিল? কোন প্রযুক্তিগত চ্যালেঞ্জ ছিল?
বর্তমান সিস্টেমে কী সীমাবদ্ধতা ছিল?]

## Decision Drivers
- [ড্রাইভার ১: যেমন — পারফরম্যান্স]
- [ড্রাইভার ২: যেমন — টিমের দক্ষতা]
- [ড্রাইভার ৩: যেমন — খরচ]

## Considered Options
1. [অপশন ১]
2. [অপশন ২]
3. [অপশন ৩]

## Decision
[কোন সিদ্ধান্ত নেওয়া হলো এবং কেন?]

## Consequences

### Positive
- [ভালো দিক ১]
- [ভালো দিক ২]

### Negative
- [খারাপ দিক ১]
- [খারাপ দিক ২]

### Risks
- [ঝুঁকি ১]
- [ঝুঁকি ২]

## References
- [লিঙ্ক, ডকুমেন্ট, ব্লগ পোস্ট ইত্যাদি]
```

### টেমপ্লেটের প্রতিটি অংশ ব্যাখ্যা

```
+------------------------------------------------------------------+
|                     ADR টেমপ্লেট ফ্লো                              |
+------------------------------------------------------------------+
|                                                                  |
|  Title ──── সংক্ষিপ্ত, বর্ণনামূলক শিরোনাম                        |
|    │                                                             |
|  Status ─── বর্তমান অবস্থা (Proposed/Accepted/Deprecated)        |
|    │                                                             |
|  Context ── কেন এই সিদ্ধান্ত নেওয়া দরকার?                       |
|    │         ├── সমস্যা কী?                                      |
|    │         ├── সীমাবদ্ধতা কী?                                   |
|    │         └── বিজনেস রিকোয়ারমেন্ট কী?                        |
|    │                                                             |
|  Decision Drivers ── কোন বিষয়গুলো সিদ্ধান্তকে প্রভাবিত করেছে?   |
|    │                                                             |
|  Options ── কী কী বিকল্প বিবেচনা করা হয়েছে?                     |
|    │         ├── Option A (বিস্তারিত)                             |
|    │         ├── Option B (বিস্তারিত)                             |
|    │         └── Option C (বিস্তারিত)                             |
|    │                                                             |
|  Decision ── চূড়ান্ত সিদ্ধান্ত + কারণ                            |
|    │                                                             |
|  Consequences ── পরিণতি                                          |
|              ├── ✅ ভালো দিক                                      |
|              ├── ❌ খারাপ দিক                                     |
|              └── ⚠️  ঝুঁকি                                       |
|                                                                  |
+------------------------------------------------------------------+
```

## 🔄 ৪. ADR লাইফসাইকেল (ADR Lifecycle)

একটি ADR তৈরি হওয়ার পর বিভিন্ন পর্যায়ের মধ্য দিয়ে যায়:

```
    ┌─────────────┐
    │  📝 Draft    │  (খসড়া — প্রাথমিক ধারণা লেখা হচ্ছে)
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 📋 Proposed  │  (প্রস্তাবিত — টিমে রিভিউয়ের জন্য শেয়ার করা হয়েছে)
    └──────┬──────┘
           │
      ┌────┴────┐
      │         │
      ▼         ▼
┌──────────┐  ┌──────────┐
│✅Accepted│  │❌Rejected│  (গৃহীত অথবা প্রত্যাখ্যাত)
└─────┬────┘  └──────────┘
      │
      │  (সময়ের সাথে...)
      │
      ├─────────────────────┐
      │                     │
      ▼                     ▼
┌────────────┐    ┌──────────────────┐
│ Deprecated │    │   Superseded     │
│ (অবচিত)    │    │ (নতুন ADR দ্বারা  │
│            │    │  প্রতিস্থাপিত)    │
└────────────┘    └──────────────────┘
```

### স্ট্যাটাস ট্রানজিশন বিস্তারিত

```
+-------------+--------------------------------------------------+
| স্ট্যাটাস    | বর্ণনা                                            |
+-------------+--------------------------------------------------+
| Draft       | ADR লেখা হচ্ছে, এখনও রিভিউয়ের জন্য প্রস্তুত নয়  |
| Proposed    | টিম রিভিউ করছে, ফিডব্যাক দিচ্ছে                   |
| Accepted    | টিম সম্মত হয়েছে, এই সিদ্ধান্ত অনুসরণ করা হবে      |
| Rejected    | টিম এই প্রস্তাব গ্রহণ করেনি                        |
| Deprecated  | এই সিদ্ধান্ত আর প্রাসঙ্গিক নয়                      |
| Superseded  | নতুন ADR এটিকে প্রতিস্থাপিত করেছে                  |
+-------------+--------------------------------------------------+
```

## 📝 ৬. বাস্তব উদাহরণ — সম্পূর্ণ ADR (Real-World ADR Examples)

---

### 🏦 ADR-001: bKash ট্রানজ্যাকশন সিস্টেমের জন্য ডেটাবেস নির্বাচন

```markdown
# ADR-001: ট্রানজ্যাকশন সিস্টেমের জন্য PostgreSQL নির্বাচন

## Status
Accepted

## Date
2024-03-15

## Context
bKash-এর ট্রানজ্যাকশন সিস্টেম বর্তমানে প্রতিদিন ৫০ মিলিয়ন+ ট্রানজ্যাকশন
প্রসেস করে। আমাদের একটি ডেটাবেস দরকার যা:

- উচ্চ ট্রানজ্যাকশন ভলিউম হ্যান্ডেল করতে পারে (TPS: 10,000+)
- ACID কমপ্লায়েন্ট (আর্থিক লেনদেনে অত্যন্ত জরুরি)
- জটিল কুয়েরি সাপোর্ট করে (রিপোর্টিং ও অ্যানালিটিক্স)
- হরাইজন্টাল স্কেলিং সম্ভব
- বাংলাদেশ ব্যাংকের রেগুলেটরি রিকোয়ারমেন্ট পূরণ করে

বর্তমান MySQL 5.7 সিস্টেম পিক আওয়ারে (ঈদ, বেতন দিবস) স্লো হয়ে যাচ্ছে।

## Decision Drivers
- পারফরম্যান্স: উচ্চ TPS (Transactions Per Second)
- ডেটা ইন্টেগ্রিটি: আর্থিক লেনদেনে শূন্য ত্রুটি
- স্কেলেবিলিটি: বাড়তে থাকা ইউজার বেস
- টিমের দক্ষতা: ডেভেলপারদের অভিজ্ঞতা
- খরচ: লাইসেন্সিং ও অপারেশনাল খরচ
- কমপ্লায়েন্স: বাংলাদেশ ব্যাংকের নিয়মাবলী

## Considered Options

### Option 1: MySQL 8.0 (আপগ্রেড)
### Option 2: PostgreSQL 16
### Option 3: Oracle Database
```

#### তুলনা টেবিল

```
+----------------------+--------------+----------------+-----------+
| বৈশিষ্ট্য             | MySQL 8.0    | PostgreSQL 16  | Oracle    |
+----------------------+--------------+----------------+-----------+
| TPS (benchmark)      | ~8,000       | ~12,000        | ~15,000   |
| ACID Compliance      | ✅ (InnoDB)   | ✅ নেটিভ       | ✅ নেটিভ  |
| JSON Support         | বেসিক        | অ্যাডভান্সড     | ভালো      |
| Partitioning         | বেসিক        | অ্যাডভান্সড     | এক্সেলেন্ট |
| Replication          | Async/Semi   | Sync/Async     | RAC       |

+----------------------+--------------+----------------+-----------+
```

```markdown

আমরা **PostgreSQL 16** নির্বাচন করেছি কারণ:

1. **উচ্চতর TPS**: MySQL-এর তুলনায় ৫০% বেশি ট্রানজ্যাকশন হ্যান্ডেল করতে পারে
2. **অ্যাডভান্সড পার্টিশনিং**: তারিখ অনুযায়ী ট্রানজ্যাকশন ডেটা পার্টিশন করা সহজ

5. **সিনক্রোনাস রেপ্লিকেশন**: আর্থিক ডেটার জন্য অত্যন্ত জরুরি

### Positive
- পিক আওয়ারেও (ঈদ/বেতন দিবস) সিস্টেম স্থিতিশীল থাকবে

### Negative

- বিদ্যমান MySQL কুয়েরি ও স্টোরড প্রসিডিওর মাইগ্রেট করতে হবে
```

### 🚗 ADR-002: Pathao রাইড ইভেন্টের জন্য মেসেজ কিউ নির্বাচন

```markdown
# ADR-002: রাইড ইভেন্ট প্রসেসিংয়ের জন্য Apache Kafka নির্বাচন

## Status
Accepted

## Date
2024-04-20

## Context
Pathao-র রাইড-শেয়ারিং সিস্টেমে প্রতি সেকেন্ডে হাজার হাজার ইভেন্ট জেনারেট হয়:
- রাইডার লোকেশন আপডেট (প্রতি ৩ সেকেন্ডে)
- রাইড রিকোয়েস্ট
- ড্রাইভার অ্যাক্সেপ্ট/রিজেক্ট
- ট্রিপ স্ট্যাটাস আপডেট
- ফেয়ার ক্যালকুলেশন
- রেটিং ও রিভিউ

বর্তমান সিস্টেমে সিনক্রোনাস API কলে ল্যাটেন্সি বেড়ে যাচ্ছে এবং
পিক আওয়ারে (সকাল ৮-১০, বিকাল ৫-৭) সিস্টেম স্লো হচ্ছে।
ঢাকা শহরে একসাথে ~৫০,০০০ অ্যাক্টিভ রাইড থাকতে পারে।

আমাদের একটি মেসেজ কিউ দরকার যা:
- উচ্চ থ্রুপুট সাপোর্ট করে (100K+ events/sec)
- ইভেন্ট রিপ্লে করার সুবিধা দেয়
- মাল্টিপল কনজিউমার সাপোর্ট করে
- ডেটা পারসিস্টেন্ট রাখে

## Decision Drivers
- থ্রুপুট: প্রতি সেকেন্ডে ১ লাখ+ ইভেন্ট
- ল্যাটেন্সি: রিয়েল-টাইম ইভেন্ট (< 100ms)
- ডিউরেবিলিটি: কোনো ইভেন্ট হারানো যাবে না
- স্কেলেবিলিটি: পিক আওয়ারে ৫x লোড হ্যান্ডেল
- ইভেন্ট রিপ্লে: অ্যানালিটিক্স ও ডিবাগিংয়ের জন্য
- অপারেশনাল কমপ্লেক্সিটি: রক্ষণাবেক্ষণ সহজ হওয়া

## Considered Options

### Option 1: RabbitMQ
### Option 2: Apache Kafka
### Option 3: Amazon SQS
```

```markdown
## Decision
আমরা **Apache Kafka** নির্বাচন করেছি কারণ:

1. **অতুলনীয় থ্রুপুট**: প্রতি সেকেন্ডে ১ মিলিয়ন+ মেসেজ হ্যান্ডেল করতে পারে

5. **পার্টিশন-বেইজড অর্ডারিং**: রাইড ID দিয়ে পার্টিশন করলে

## Consequences

### Positive
- রিয়েল-টাইম ইভেন্ট প্রসেসিং সম্ভব হবে
- একই ইভেন্ট ৫+ সার্ভিস কনজিউম করতে পারবে (fan-out)
- ইভেন্ট সোর্সিং প্যাটার্ন ইমপ্লিমেন্ট করা যাবে

### Negative
- Kafka-র অপারেশনাল কমপ্লেক্সিটি বেশি (ZooKeeper/KRaft ম্যানেজমেন্ট)
- টিমকে Kafka ট্রেনিং দিতে হবে (৩-৪ সপ্তাহ)
- ইনফ্রাস্ট্রাকচার খরচ RabbitMQ-এর চেয়ে বেশি

### Risks
- Kafka ক্লাস্টার ম্যানেজমেন্টে ভুল হলে ডেটা লস হতে পারে
  (মিটিগেশন: AWS MSK ব্যবহার করা যেতে পারে)
- কনজিউমার ল্যাগ বেড়ে গেলে রিয়েল-টাইমনেস কমবে
```

```javascript
// services/ride-event-producer.js — ADR-002 অনুযায়ী Kafka ইভেন্ট প্রোডিউসার

const { Kafka } = require('kafkajs');

const kafka = new Kafka({
    clientId: 'pathao-ride-service',
    brokers: ['kafka-1.pathao.internal:9092', 'kafka-2.pathao.internal:9092'],
    ssl: true,
});

const producer = kafka.producer();

// রাইড ইভেন্ট পাবলিশ — রাইড ID দিয়ে পার্টিশনিং
async function publishRideEvent(rideId, eventType, payload) {
    await producer.send({
        topic: 'ride-events',
        messages: [{
            key: rideId,   // একই রাইডের ইভেন্ট একই পার্টিশনে যাবে
            value: JSON.stringify({
                eventType,  // RIDE_REQUESTED, DRIVER_ACCEPTED, TRIP_STARTED, etc.
                rideId,
                timestamp: Date.now(),
                payload,
            }),
        }],
    });
}

// ড্রাইভার লোকেশন আপডেট — আলাদা টপিক, উচ্চ ফ্রিকোয়েন্সি
async function publishLocationUpdate(driverId, lat, lng) {
    await producer.send({
        topic: 'driver-locations',
        messages: [{
            key: driverId,
            value: JSON.stringify({ driverId, lat, lng, timestamp: Date.now() }),
        }],
    });
}

module.exports = { publishRideEvent, publishLocationUpdate };
```

### 🛒 ADR-003: Daraz প্রোডাক্ট ক্যাটালগের জন্য API স্টাইল নির্বাচন

```markdown
# ADR-003: প্রোডাক্ট ক্যাটালগ API-এর জন্য GraphQL নির্বাচন

## Status
Accepted

## Date
2024-05-10

## Context
Daraz বাংলাদেশের প্রোডাক্ট ক্যাটালগে ৫০ লাখ+ প্রোডাক্ট আছে। বর্তমান REST
API-তে কিছু সমস্যা আছে:

1. **Over-fetching**: প্রোডাক্ট লিস্ট পেজে শুধু নাম, ছবি ও দাম দরকার,
   কিন্তু API পুরো প্রোডাক্ট অবজেক্ট (৫০+ ফিল্ড) পাঠায়
2. **Under-fetching**: প্রোডাক্ট ডিটেইল পেজে প্রোডাক্ট + রিভিউ + সেলার
   + শিপিং তথ্য দরকার — ৪টি আলাদা API কল করতে হয়
3. **N+1 সমস্যা**: ক্যাটাগরি পেজে ২০টি প্রোডাক্ট + প্রতিটির সেলার
   তথ্য পেতে ২১টি API কল
4. **মোবাইল পারফরম্যান্স**: বাংলাদেশের ধীর নেটওয়ার্কে (2G/3G) অনেক API
   কল করলে UX খারাপ হয়

এপ্লিকেশনের ক্লায়েন্ট: Android App, iOS App, ওয়েব, সেলার ড্যাশবোর্ড।

## Decision Drivers
- ক্লায়েন্ট ফ্লেক্সিবিলিটি: বিভিন্ন ক্লায়েন্টে ভিন্ন ডেটা দরকার
- নেটওয়ার্ক এফিশিয়েন্সি: বাংলাদেশের ধীর নেটওয়ার্কে কম কল
- ডেভেলপার এক্সপেরিয়েন্স: ফ্রন্টএন্ড ডেভেলপারদের সুবিধা
- ক্যাশিং: API রেসপন্স ক্যাশ করার সুবিধা
- পারফরম্যান্স: ব্যাকএন্ড লোড

## Considered Options

### Option 1: REST API (বিদ্যমান, উন্নত)
### Option 2: GraphQL
### Option 3: gRPC
```

#### তুলনা টেবিল

```
+---------------------------+-------------+---------------+----------+
| বৈশিষ্ট্য                  | REST API    | GraphQL       | gRPC     |
+---------------------------+-------------+---------------+----------+
| ক্লায়েন্ট ফ্লেক্সিবিলিটি  | নিম্ন       | উচ্চ          | মাঝারি   |

| ব্রাউজার সাপোর্ট           | ✅ নেটিভ     | ✅ লাইব্রেরি   | ❌ সীমিত |

| টিমের অভিজ্ঞতা             | ৫/৫         | ২/৫           | ১/৫     |
+---------------------------+-------------+---------------+----------+
```

```markdown
## Decision
আমরা প্রোডাক্ট ক্যাটালগের জন্য **GraphQL** নির্বাচন করেছি:

1. **Over-fetching সমাধান**: ক্লায়েন্ট শুধু যা দরকার তা-ই রিকোয়েস্ট করবে
2. **Under-fetching সমাধান**: একটি কুয়েরিতে প্রোডাক্ট + রিভিউ + সেলার পাওয়া যাবে
3. **মোবাইল অপটিমাইজেশন**: বাংলাদেশের ধীর নেটওয়ার্কে কম API কল = ভালো UX
4. **মাল্টি-ক্লায়েন্ট**: Android, iOS, ওয়েব — প্রত্যেকে নিজের মতো ডেটা ফেচ করবে

**তবে**: পেমেন্ট, অথেনটিকেশন ও সেলার API-তে REST বহাল থাকবে।
শুধুমাত্র প্রোডাক্ট ক্যাটালগেই GraphQL ব্যবহার হবে।

## Consequences

### Positive
- প্রোডাক্ট লিস্ট পেজে ডেটা ট্রান্সফার ৬০% কমবে
- মোবাইল অ্যাপের রেসপন্স টাইম উল্লেখযোগ্যভাবে কমবে
- ফ্রন্টএন্ড ডেভেলপাররা ব্যাকএন্ডের উপর কম নির্ভরশীল হবে
- নতুন ফিচার দ্রুত ডেভেলপ করা যাবে

### Negative
- HTTP ক্যাশিং জটিল হবে (সমাধান: Persisted Queries + CDN)
- ব্যাকএন্ডে N+1 কুয়েরি সমস্যা হতে পারে (সমাধান: DataLoader)
- টিমকে GraphQL শিখতে হবে (২-৩ সপ্তাহ ট্রেনিং)
- দুটি API স্ট্যাটেজি (REST + GraphQL) মেইনটেইন করতে হবে

### Risks
- GraphQL কুয়েরি কমপ্লেক্সিটি আক্রমণ (মিটিগেশন: কুয়েরি ডেপথ লিমিট)
- ক্যাশিং চ্যালেঞ্জ (মিটিগেশন: Apollo Server ক্যাশিং + Redis)

## References
- "Production Ready GraphQL" by Marc-Andre Giroux
- Shopify GraphQL Case Study
- Daraz Internal API Audit Report Q1 2024
```

#### সংশ্লিষ্ট কোড উদাহরণ — GraphQL Schema ও Resolver (JavaScript)

```javascript
// schema/product.graphql — ADR-003 অনুযায়ী GraphQL স্কিমা

const typeDefs = `
  type Product {
    id: ID!
    name: String!
    nameBn: String          # বাংলা নাম
    price: Float!
    discountPrice: Float
    images: [String!]!
    category: Category!
    seller: Seller!
    reviews(limit: Int = 5): [Review!]!
    rating: Float
    shippingInfo: ShippingInfo!
  }

  type Query {
    # প্রোডাক্ট লিস্ট — ক্লায়েন্ট যা দরকার শুধু তাই রিকোয়েস্ট করবে
    products(
      category: ID
      search: String
      limit: Int = 20
      offset: Int = 0
    ): ProductConnection!

    # সিঙ্গেল প্রোডাক্ট — ডিটেইল পেজের জন্য
    product(id: ID!): Product
  }
`;

// resolvers/product.js — DataLoader দিয়ে N+1 সমাধান
const DataLoader = require('dataloader');

const sellerLoader = new DataLoader(async (sellerIds) => {
    const sellers = await Seller.find({ _id: { $in: sellerIds } });
    return sellerIds.map(id => sellers.find(s => s._id.equals(id)));
});

const resolvers = {
    Product: {
        seller: (product) => sellerLoader.load(product.sellerId),
        reviews: (product, { limit }) =>
            Review.find({ productId: product.id }).limit(limit),
    },
};
```

```
❌ ভুল: ADR কোনো আলাদা টুলে (Confluence) রাখা
✅ সঠিক: কোড রিপোজিটরিতে Markdown ফাইল হিসেবে রাখা

❌ ভুল: শুধু সিদ্ধান্ত লেখা, প্রেক্ষাপট না লেখা
✅ সঠিক: কেন এই সিদ্ধান্ত নেওয়া হলো তা বিস্তারিত লেখা

❌ ভুল: Accepted ADR পরিবর্তন করা
✅ সঠিক: নতুন ADR তৈরি করে পুরানোটিকে Supersede করা

❌ ভুল: প্রতিটি ছোট সিদ্ধান্তের জন্য ADR লেখা
✅ সঠিক: শুধু গুরুত্বপূর্ণ আর্কিটেকচার সিদ্ধান্তের জন্য ADR লেখা

❌ ভুল: ADR লিখে ভুলে যাওয়া
✅ সঠিক: নিয়মিত রিভিউ করা, পুরানো ADR Deprecate করা
```

## 📝 ১৩. দ্রুত শুরু করুন (Quick Start)

### ৫ মিনিটে আপনার প্রথম ADR

```bash
# ১. প্রোজেক্টে ADR ফোল্ডার তৈরি
mkdir -p docs/adr

# ২. টেমপ্লেট ফাইল তৈরি
cat > docs/adr/template.md << 'EOF'
# ADR-[NUMBER]: [TITLE]

## Status
Proposed

## Date
[YYYY-MM-DD]

## Context
[সমস্যা বা পরিস্থিতি বর্ণনা করুন]

## Decision Drivers
- [ড্রাইভার ১]
- [ড্রাইভার ২]

## Considered Options
1. [অপশন ১]
2. [অপশন ২]

## Decision
[কোন সিদ্ধান্ত নেওয়া হলো এবং কেন]

## Consequences
### Positive
- [ভালো দিক]

### Negative
- [খারাপ দিক]

### Risks
- [ঝুঁকি]
EOF

# ৩. প্রথম ADR তৈরি
cp docs/adr/template.md docs/adr/0001-initial-tech-stack.md

# ৪. README তৈরি
echo "# Architecture Decision Records" > docs/adr/README.md
echo "| ADR | Title | Status | Date |" >> docs/adr/README.md
echo "|-----|-------|--------|------|" >> docs/adr/README.md

# ৫. Git-এ কমিট
git add docs/adr/
git commit -m "docs(adr): initialize ADR directory and template"
```

### PHP প্রোজেক্টে ADR জেনারেটর স্ক্রিপ্ট

```php
<?php
// scripts/create-adr.php — সহজ ADR জেনারেটর

$adrDir = __DIR__ . '/../docs/adr';

if (!is_dir($adrDir)) {
    mkdir($adrDir, 0755, true);
}

// পরবর্তী ADR নম্বর বের করা
$existingFiles = glob($adrDir . '/[0-9]*.md');
$nextNumber = count($existingFiles) + 1;
$paddedNumber = str_pad($nextNumber, 4, '0', STR_PAD_LEFT);

// CLI থেকে শিরোনাম নেওয়া
$title = $argv[1] ?? 'untitled';
$slug = strtolower(preg_replace('/[^a-zA-Z0-9]+/', '-', $title));
$filename = "{$paddedNumber}-{$slug}.md";
$date = date('Y-m-d');

$template = <<<MD
# ADR-{$paddedNumber}: {$title}

## Status
Proposed

## Date
{$date}

## Context
[বর্ণনা করুন]

## Decision Drivers
- [ড্রাইভার ১]

## Considered Options
1. [অপশন ১]
2. [অপশন ২]

## Decision
[সিদ্ধান্ত ও কারণ]

## Consequences
### Positive
- [ভালো দিক]

### Negative
- [খারাপ দিক]
MD;

file_put_contents("{$adrDir}/{$filename}", $template);
echo "✅ ADR তৈরি হয়েছে: docs/adr/{$filename}\n";

// Usage: php scripts/create-adr.php "Use Redis for session caching"
```
